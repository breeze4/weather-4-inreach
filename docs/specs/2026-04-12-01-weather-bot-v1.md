# Weather Bot v1 — InReach Weather Service

## Problem Statement

The wx2inreach.com service that provided weather forecasts via Garmin InReach satellite messenger is dead. When in the backcountry with no cell service, there's no way to get a weather forecast by texting a keyword from the InReach device. The user needs a self-hosted replacement that's scrappy and reliable enough to depend on for real trips.

## Solution

A self-hosted Python service that polls a dedicated Gmail inbox (configured via `GMAIL_USER`) for InReach messages, fetches weather from the NWS API, formats the forecast to fit the 160-character InReach message limit using Gemini 3.1 Flash, and sends the result as a new email to the user's InReach address.

Two commands supported:
- `wx now` — current 24-hour forecast (today/tonight/tomorrow day/tomorrow night) with temps, precip %, and conditions
- `wx week` — 7-day forecast, packing as many day/night periods as possible into 160 chars, truncating from the end

## User Stories

1. As a backcountry traveler, I want to send "wx now" from my InReach and receive a 24-hour weather forecast, so that I can make decisions about today's route and camp
2. As a backcountry traveler, I want to send "wx week" from my InReach and receive a multi-day forecast, so that I can plan whether to stay, bail, or adjust my itinerary
3. As a backcountry traveler, I want the forecast to include precipitation percentage, so that I can distinguish between "20% chance of rain" and "80% chance of rain" at the same temperature
4. As a backcountry traveler, I want the response to arrive within a few minutes, so that I'm not waiting around burning satellite time
5. As a backcountry traveler, I want error messages when something breaks, so that I know to retry rather than wondering if my message was lost
6. As the service operator, I want the service to retry NWS failures automatically for 20 minutes, so that transient outages don't require manual intervention
7. As the service operator, I want malformed or non-InReach emails silently ignored, so that spam or accidental emails don't trigger responses
8. As the service operator, I want the service to run unattended on beebaby, so that it works while I'm in the field with no access to fix it

## Data Flow

1. **Poll**: Long-running process checks Gmail IMAP every 2 minutes for unread emails
2. **Filter**: Match subject `inReach message from Jane Doe`. Ignore everything else.
3. **Parse command**: Extract `wx now` or `wx week` from email body (case-insensitive). Ignore emails with unrecognized commands.
4. **Parse location**: Extract lat/lon from the line `Jane Doe sent this message from: Lat XX.XXXXXX Lon -XXX.XXXXXX`
5. **Fetch weather**: Call NWS points API (`/points/{lat},{lon}`) to get the forecast URL, then fetch the 12-hourly forecast
6. **Format**: Send NWS forecast data + command type to Gemini 3.1 Flash with a prompt template defining the output format and constraints. Validate the response: must be ≤160 chars and match the expected format. Retry Gemini up to 3 times if validation fails.
7. **Send**: Compose a new email via SMTP to the InReach device via the Garmin reply page with the formatted forecast as the body
8. **Mark read**: Mark the original Gmail message as read so it's not reprocessed

### NWS Retry Flow

If the NWS API fails at step 5:
1. Immediately send an error message to the InReach: terse message indicating NWS is down, retrying for 20 minutes
2. Retry NWS every 2 minutes for up to 10 attempts
3. If any retry succeeds, send the formatted forecast
4. If all 10 retries fail, give up silently (the initial error message already informed the user)

## Behavior

- Command parsing is case-insensitive (`wx now`, `Wx Now`, `WX NOW` all work)
- Only emails with subject exactly `inReach message from Jane Doe` are processed
- Emails without a recognized command (`wx now` or `wx week`) are silently ignored
- Lat/lon is always present in the email body — no fallback parsing needed
- `wx now` returns ~4 periods (today, tonight, tomorrow, tomorrow night) with temp, precip %, and abbreviated condition
- `wx week` packs as many day/night periods as fit into 160 chars, truncating from the end rather than sending multiple messages
- Replies are new outbound emails (not IMAP replies) to the InReach address
- Processed emails are marked as read
- The service is a long-running process with a 2-minute sleep loop (not a cron job)

### Message Format

Messages must be ≤160 characters. Gemini 3.1 Flash generates the formatted output from NWS data using a prompt template. The prompt should include example output formats based on the old service:

**wx now example** (~4 periods, day/night pairs with precip % when relevant):
```
Td Sun 74 Tn Clr 57 Wed PCldy 68 WedN Clr 52
```

**wx now with precip example**:
```
Td RnShw 58% 57 Tn RnShw 39% 41 Wed ChRn 40% 55 WedN MCldy 18% 43
```

**wx week example** (as many days as fit):
```
Sun 74 TnClr57 Mon 80 Clr59 Tue 81 Clr60 Wed 85 Clr62 Thu 82 Clr59 Fr 80
```

The exact format will be defined in the Gemini prompt template. The validation step checks:
1. Total length ≤160 characters
2. Contains recognizable day abbreviations and temperatures

If validation fails, Gemini is called again with the previous output and correction instructions (up to 3 total attempts).

## Modules

- **Email Poller**: Connects to Gmail via IMAP, fetches unread messages, filters by subject, marks as read after processing
  - Role: **defines** the message ingestion interface
  - Interface: yields parsed email objects (subject, body, message ID)
  - Test: no

- **InReach Parser**: Extracts command type and lat/lon from email body
  - Role: **defines** the parsed request structure (command enum + coordinates)
  - Interface: takes raw email body, returns command + lat/lon or None for unrecognized emails
  - Test: yes — deterministic parsing, easy to unit test against the known email format

- **NWS Client**: Fetches forecast from NWS API given lat/lon
  - Role: **consumes** lat/lon, **defines** raw forecast data structure
  - Interface: takes lat/lon, returns forecast periods (name, temp, precip, shortForecast)
  - Test: no (external API)

- **Forecast Formatter**: Sends NWS data to Gemini 3.1 Flash, validates output, retries if needed
  - Role: **consumes** forecast data, **defines** the final message string
  - Interface: takes command type + forecast periods, returns ≤160 char string
  - Test: yes — validate the validation logic (char count, format check) with known inputs

- **Reply Sender**: Sends formatted forecast via SMTP to the InReach address
  - Role: **consumes** the final message string
  - Interface: takes message string, sends email
  - Test: no

- **Retry Manager**: Tracks NWS retry state for failed requests (up to 10 retries over 20 minutes)
  - Role: **defines** retry lifecycle
  - Interface: enqueue a failed request, check/execute pending retries each poll cycle
  - Test: yes — state machine logic

- **Main Loop**: Orchestrates poll → parse → fetch → format → send cycle, manages retry queue
  - Role: **consumes** all other modules
  - Interface: entry point, runs forever
  - Test: no

## Resolved Decisions

- **Two commands only**: `wx now` and `wx week` — no `nws6`, `vc`, `loc`, or metric flags. Keeps v1 minimal.
- **NWS only, no VisualCrossing**: User is US-only for the foreseeable future. Eliminates a dependency and API key.
- **Gemini 3.1 Flash for formatting**: Instead of a deterministic abbreviation engine, use an LLM to compress NWS forecast text into ≤160 chars. Adds a dependency but avoids maintaining an abbreviation dictionary. Format validation loop catches LLM mistakes.
- **IMAP + App Password over Gmail API/OAuth**: Simpler setup, no token refresh, no Google Cloud project. User already has the Gmail account and password.
- **Long-running process over systemd timer**: Retry state lives in memory. Systemd restarts on crash.
- **Single message, truncate from end**: For `wx week`, pack as many periods as fit in 160 chars rather than sending 2 messages. User prefers 1 message with 5-6 days over 2 messages with 7.
- **New email, not reply**: InReach reply-to is the device's `@inreach.garmin.com` address. Compose a fresh outbound email rather than replying to the Garmin no-reply address.
- **Filter by subject line**: Only process emails with subject `inReach message from Jane Doe`. Silently ignore everything else.
- **No Garmin page scraping**: Lat/lon is inline in the email body. The Garmin share link exists but is not needed.
- **Poll every 2 minutes**: Balance between responsiveness and not hammering Gmail.
- **Deploy to beebaby**: Systemd user service, rsync deploys. Same model works on a GCP e2-micro VM later.

## Testing Decisions

- **InReach Parser**: Unit tests against the known email format (the sample .eml file serves as a fixture)
- **Forecast Formatter validation logic**: Unit tests for the ≤160 char check and format validation
- **Retry Manager**: Unit tests for the state machine (enqueue, retry counting, expiry)
- **No integration tests for v1**: External dependencies (Gmail, NWS, Gemini) are tested manually by sending a real InReach message before the first trip
- **Manual end-to-end test required before field use**: Send a real `wx now` and `wx week` from the InReach device and verify the response arrives

## Dependency Strategy

- **NWS API**: Mock boundary — no API key, free, but external. Retry with backoff on failure.
- **Gemini 3.1 Flash**: Mock boundary — API key in `.env`. Validation loop handles bad output.
- **Gmail IMAP/SMTP**: Mock boundary — App Password in `.env`. No test substitute; tested manually.

## Out of Scope

- VisualCrossing or any non-NWS weather source
- International locations
- Multiple users or devices
- `nws6`, `vc`, `loc`, metric, or any command flags beyond `wx now` / `wx week`
- Multi-message responses (splitting forecast across 2+ messages)
- OAuth / Gmail API (using IMAP + App Password)
- Web UI, dashboard, or monitoring
- Automated CI/CD pipeline
- Wind data in forecasts
